## Step 6 — Feedback (direct, no false credit)

**Coach message:**

> That is wrong. The actual output is:
>
> ```
> {'click': ['button_a', 'button_b', 'button_c'], 'load': ['page_1']}
> ```
>
> Your prediction ("each call gets a fresh `history={}`") is wrong in a precise way:
> **`history={}` is evaluated exactly once — when `def log_event(...)` is first
> executed, at *definition time*.** That single dict object is stored on the function
> as `log_event.__defaults__[0]` and is reused by every call that omits the `history`
> argument.
>
> The machine timeline:
>
> 1. `def log_event(...)` runs → one `{}` dict is created and stored in `__defaults__`.
> 2. Call 1 (`button_a`) → `history` is rebound to that **same dict**; `click` list
>    created; `button_a` appended. Dict is now `{'click': ['button_a']}`.
> 3. Call 2 (`button_b`) → `history` rebound to the **same dict again**; `button_b`
>    appended. Dict: `{'click': ['button_a', 'button_b']}`.
> 4. Call 3 (`page_1`) → `load` key created; `page_1` appended. Dict:
>    `{'click': ['button_a', 'button_b'], 'load': ['page_1']}`.
> 5. Call 4 (`button_c`) → `button_c` appended to `click`. Dict:
>    `{'click': ['button_a', 'button_b', 'button_c'], 'load': ['page_1']}`.
>    This is what gets printed.
>
> The underlying model: **default expressions run once, at definition time — not per call.**
> This is not a quirk; it follows directly from the execution rule (statements run
> top-to-bottom exactly once, when encountered). The `def` statement runs once; the `={}`
> runs with it. After that, the function object holds that object and hands it out.
>
> The catalog label for this gap: **"mutable default argument re-initialized per call."**


**Why (coach's ground-truth explanation, for the record):**

1. **`store={}` — one dict, shared.** `make_logger` is defined once; its `store={}` is
   evaluated once. Both `make_logger("x")` calls operate on the same dict.
2. **`store.setdefault("x", [])` — one list.** The first call creates `store["x"] = []`.
   The second call finds the key already present and does nothing. Both `r1` and `r2`'s
   `record` closures close over the same `store` dict (the default) and the same `tag`
   (`"x"`) — so `store[tag]` in both closures refers to the **same list object**.
3. **`r1(10)` → appends `10`; `r2(20)` → appends `20` to the same list.** List is
   `[10, 20]`.
4. **`log = r1(30)`** → appends `30`; list is `[10, 20, 30]`; `record` returns
   `store["x"]`, which IS that list → `log` is bound to `[10, 20, 30]`.
5. **`print(log)`** → `[10, 20, 30]`.
6. **`print(log is r2(40))`** → `r2(40)` appends `40`, list is now `[10, 20, 30, 40]`,
   returns `store["x"]`. `log` still points at the **same list object** (mutation, not
   rebinding). So `log is r2(40)` is `True`.
7. **`print(len(log))`** → `log` is the same object, now with 4 elements → `4`.

Three mechanisms in sequence: mutable-default dict → aliasing of the inner list →
closure reading `store[tag]` at call time. A learner who predicts `[10, 20, 30]` but
says `False` for line 2 has aliasing but not identity; one who says `len == 3` has not
tracked that `r2(40)` mutated the same object `log` points at.

### 1. Runner path resolution — ambiguous for non-`~/.claude` installs

install), the path is `<repo-root>/programming-mastery/runtime/python/runner.py`."
Or: "resolve against the directory containing SKILL.md, however that directory is
accessed."
